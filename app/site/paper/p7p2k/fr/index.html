No. 06 · Technical

Le harnais bien conçu

Comment un petit ensemble de principes transforme un agent de codage en équipe de livraison qui expédie sans risque à la vitesse du « vibe coding ».

Abstract. Le rôle du gouvernement est d’avancer à la vitesse des personnes qu’il sert, avec le même soin pour la sécurité qu’une banque ou un hôpital. Ce livre blanc explique comment. Il présente les cinq principes qui transforment un agent de codage en équipe de livraison. Un fichier contrat que l’agent lit en premier. Des fichiers de compétences qui chargent l’expertise à la demande. Des hooks qui se déclenchent avant que l’agent ne s’arrête. La cybersécurité tissée dans le travail plutôt qu’ajoutée à la fin. Et deux sortes de vérifications pour deux sortes de questions. Mis ensemble, ces principes produisent un harnais opérationnel : un dispositif dans lequel une personne de tout horizon peut s’asseoir, faire équipe avec un agent et livrer une application qui satisfait les normes qu’un auditeur poserait. Le résultat est un logiciel de calibre entreprise produit à la vitesse d’un prototype de hackathon. Ce livre fait partie des Velocity Papers et explique un morceau de comment nous y arrivons.
La vélocité est l’objectif du gouvernement. Nous devons avancer à la vitesse des personnes que nous servons, et nous devons les protéger en chemin. Les Velocity Papers expliquent comment. Celui-ci, le livre 8, traite d’un seul morceau de la réponse. Un morceau assez petit pour qu’une personne le tienne entier en tête, et assez utile pour qu’une personne de tout horizon puisse s’asseoir, faire équipe avec un agent de codage et livrer une application qu’une industrie réglementée accepterait. Nous appelons ce morceau le harnais.


## §01 L’objectif, c’est la vélocité

Moderniser un parc applicatif à l’échelle d’une province est un problème de cadence. Le livre 1 pose la taille du problème. Le livre 8 traite du rythme auquel le travail doit avancer pour entamer ce parc. Ingénieurs, analystes de politiques, agents de programmes, étudiants, stagiaires : le travail doit être fait par tous ces gens, plutôt que par le seul petit nombre d’ingénieurs séniors qu’un gouvernement peut se permettre d’embaucher. Le harnais rend cela possible. Il encode les normes que les ingénieurs séniors appliqueraient, de sorte que les normes voyagent avec le travail au lieu de vivre dans la tête d’une seule personne.

"Les normes doivent voyager avec le travail, plutôt que vivre dans la tête d’une seule personne." · Livre 8 : Le harnais bien conçu


## §02 Ce qu’est un harnais, et ce qu’il n’est pas

Un harnais est l’environnement de travail dans lequel l’agent vient s’installer. C’est une arborescence de dossiers avec quelques fichiers précis, plus un peu d’automatisation. Quand l’agent l’ouvre, le harnais a déjà pré-décidé les choses que le projet ne veut pas redécider à chaque construction : comment on protège les routes, comment on assainit le contenu, comment on exécute les migrations, comment on verrouille les livraisons, à quoi devrait ressembler une page finie. L’agent n’a pas à se souvenir de ces décisions. C’est le harnais qui s’en souvient.

Un harnais n’est pas un gabarit de démarrage. Un gabarit est ce que l’on copie. Le harnais est la discipline qui l’entoure. On peut mettre un harnais autour de presque n’importe quel gabarit, et l’on peut remplacer le gabarit à l’intérieur sans changer le harnais. Le harnais est la partie qui survit à une refonte.


## §03 Principe 1 : le CLAUDE.md est le contrat

Chaque harnais s’ouvre sur un fichier nommé CLAUDE.md. Il vit à la racine du projet et c’est la première chose que l’agent lit à chaque session. C’est le contrat. Il énonce, en une page ou moins, à quoi sert le projet, qui il sert, à quoi ressemble le bon travail, ce que l’agent peut modifier et ce qu’il ne doit jamais toucher. Les nouveaux développeurs le lisent pour la même raison : il leur dit quels compromis l’équipe a déjà tranchés.

Ce qui vit dans un CLAUDE.md Une page. À quoi sert le projet. Qui il sert. Les cinq contraintes. La carte des dossiers. Les conventions. La liste des choses à ne pas faire. Les commandes qu’un contributeur exécutera. En cas de doute, le CLAUDE.md est ce que l’on lit.

Le contrat doit être court, parce que l’agent le lit à chaque fois. Il doit être précis, parce que l’agent le prend au pied de la lettre. Et il doit être honnête sur ce qu’il ne couvre pas, parce que l’agent comblera les vides avec ce que sa formation lui a appris. Un bon CLAUDE.md ferme les vides qui comptent pour ce projet précis, et fait confiance à l’entraînement général de l’agent pour le reste.


## §04 Principe 2 : les compétences chargent à la demande

Si le CLAUDE.md est le contrat, les fichiers de compétences sont les modes opératoires. Un fichier de compétences est un petit dossier avec un seul rôle : voici comment cette équipe écrit une migration de base de données, ou voici comment cette équipe fait une revue de sécurité. Chacun porte un nom, une description et une suite d’étapes. Ils sont écrits pour qu’un agent puisse les lire au moment du besoin.

L’agent ne charge pas toutes les compétences en même temps. Il choisit celle dont il a besoin quand il en a besoin. Cela compte, parce que la capacité de lecture d’un agent, comme celle d’un lecteur, est finie. Charger le mode opératoire de revue de sécurité alors que le travail consiste à renommer un fichier évincerait le reste du contexte du projet. Les compétences restent donc dans leurs tiroirs jusqu’à ce qu’un travail les appelle.

Pourquoi une compétence est un dossier, et non un paragraphe. Une compétence porte un fichier d’instructions, une méthodologie, un gabarit de sortie, et parfois quelques scripts auxiliaires. En les regroupant comme un dossier, un contributeur peut lire la compétence complète de la même manière qu’un agent, la modifier au même endroit et la versionner d’une seule pièce.


## §05 Principe 3 : les hooks sont des portes automatiques

Le troisième principe est celui qui garde l’agent honnête. Un hook est un petit programme qui s’exécute à un moment qui compte : avant que l’agent ne déclare le travail fini, avant qu’une phase n’avance, avant que le code ne soit committé. Le hook lit le travail et répond à une question oui ou non. La migration est-elle idempotente ? Les tests requis sont-ils présents ? La prose passe-t-elle l’audit de style ? Si la réponse est non, on demande à l’agent de corriger.

Les hooks sont déterministes. Ils s’exécutent à chaque fois. On ne leur demande pas leur avis. C’est ce qui donne ses dents au harnais. Un agent pressé, ou poussé par une échéance, ne peut pas négocier avec un hook comme il pourrait amener un relecteur humain fatigué à signer.


## §06 Principe 4 : la cybersécurité est de premier rang

Le quatrième principe est une posture. La cybersécurité fait partie de la construction. Ce n’est pas une revue que l’on planifie après que le code est fini. Le harnais a les normes cyber écrites dans le contrat, des fichiers de compétences pour les appliquer, et des hooks qui vérifient que l’application a bien été faite. Quand l’agent signale fini, la cyber a déjà été faite.

La preuve visible de ce principe est le tableau périodique ci-dessous. Chaque tuile est un contrôle que le harnais maîtrise par défaut. Authentification. Autorisation. Jetons et cookies. CSRF. Injection et cross-site scripting. En-têtes de sécurité. Limites de débit. Politique réseau. Sécurité des téléversements. Journaux et audit. Secrets et cryptographie. Posture de base de données. Hygiène de la chaîne d’approvisionnement. Résilience. Tests et évaluations. Sécurité de l’IA. Seize catégories. Chaque tuile renvoie à un fichier du projet, et non à une diapositive.


## §07 Principe 5 : deux sortes de vérifications

Le logiciel pose deux sortes de questions. Le test passe-t-il ? La dépendance est-elle sur la liste CVE ? La migration s’exécute-t-elle deux fois sans changer le schéma ? Ces questions ont une seule bonne réponse. Un petit script peut y répondre. On exécute le script deux fois de la même manière, on obtient la même réponse deux fois. Ce sont les vérifications déterministes.

Le modèle de menaces est-il complet ? La prose est-elle compréhensible pour les personnes qui doivent la lire ? La conception résiste-t-elle à la façon dont un attaquant l’aborderait réellement ? Ces questions demandent du jugement. Un petit script ne peut pas y répondre. Le harnais les exécute par une vérification probabiliste : un agent à qui l’on a confié le travail et une méthode, et qui rédige une revue défendable.

Pourquoi les deux et pas une Deux pour une raison. Les vérifications déterministes donnent le plancher anti-régression. On ne les fait pas plier. Les vérifications probabilistes donnent la lecture experte. Elles attrapent ce qu’un script ne sait pas formuler. Si l’on laisse tomber l’une ou l’autre, le harnais fuit.


## §08 Ce qu’il produit : résultats d’entreprise à vitesse de hackathon

Mettez les cinq principes côte à côte et vous tenez une recette. Un contrat que l’agent lit d’abord. Des modes opératoires que l’agent tire à la demande. Des portes qui se déclenchent automatiquement. La sécurité comme fil traversant le travail, plutôt qu’une passe finale. Deux sortes de vérifications pour deux sortes de questions. Aucun de ces éléments n’est nouveau en soi. La nouveauté est qu’ils sont conçus pour fonctionner ensemble, autour d’un agent de codage, de sorte que l’agent peut avancer vite sans laisser les normes derrière.

"Nous avons fait de la cybersécurité une préoccupation de premier rang dans le harnais, tout en conservant la vitesse d’un prototype vibe-codé. Les deux ne s’opposent pas quand les normes voyagent avec le travail." · Livre 8 : Le harnais bien conçu

La preuve que cela fonctionne vient de l’Académie d’IA. Dans la dernière cohorte du Niveau 3, cent étudiants ont travaillé aux côtés d’agents de codage pendant six jours. Ils venaient autant du côté politiques et programmes que de l’ingénierie. Ensemble, ils ont produit plus de cinq cents applications opérationnelles. Chacune a franchi la revue cyber, la vérification d’accessibilité et la norme de présentation à la première version. Avant ces travaux, le repère pour les nouvelles applications gouvernementales sur ces seuils tournait autour de quarante pour cent.


## §09 La suite : des agents qui vivent dans l’application

Aujourd’hui, le harnais tourne dans le terminal. L’agent siège à côté de l’ingénieur ; les hooks se déclenchent sur le chemin de la livraison. Demain, le même agent passe à l’intérieur de l’application. Il surveille le système en marche, rafraîchit une dépendance qui vient d’atterrir sur la liste CVE, ouvre une pull request quand une configuration dérive, appelle un humain quand une découverte sort de son autorité. L’application commence à veiller sur elle-même.

C’est le sujet du prochain livre de la collection. Le harnais dans ce livre explique comment nous construisons le nouveau code en toute sûreté. Le prochain livre explique comment le code en production veille sur lui-même après livraison. Les deux moitiés répondent à la même question : comment une province modernise-t-elle un parc de quarante ans au rythme que le public attend, tout en protégeant le public tout du long ? Les Velocity Papers sont la réponse, par tranches. Restez à l’écoute pour la prochaine.


## §10 Annexe : les détails techniques, par ordre d’importance

Si les principes répondent au comment du harnais, cette annexe répond au quoi qui se trouve sous chaque principe. L’ordre n’est pas celui du code. C’est l’ordre dans lequel un attaquant, un auditeur ou un ingénieur d’astreinte se promènerait, en se demandant qu’est-ce qui est en place pour cette préoccupation, et pourquoi est-ce que cela compte ?

1. Qui peut faire quoi : authentification et autorisation. La connexion passe par OAuth 2.0 ou OpenID Connect, contre Google ou Microsoft Entra ID. Le jeton de session est un JWT signé RS256 dans un cookie httpOnly ; le jeton de rafraîchissement tourne à chaque usage et révoque l’ancien. Les rôles forment une hiérarchie à six niveaux, appliquée à la fois par le middleware et par une contrainte CHECK en base de données. Pourquoi c’est important : la mauvaise personne qui fait la bonne chose reste une compromission. L’identité et le rôle sont la première réponse à toute autre question.

2. Comment l’application se souvient de vous : jetons, cookies, CSRF. Les jetons vivent dans des cookies httpOnly que le JavaScript ne lit jamais. Une double soumission CSRF vérifie que les requêtes viennent du même site que la page. Le client n’écrit jamais dans le stockage local. Pourquoi c’est important : si un jeton fuit, un attaquant peut agir en votre nom. Garder la surface aussi petite que possible est la défense la moins chère de cette liste.

3. Empêcher l’entrée malveillante de s’exécuter : injection et XSS. Chaque fragment HTML rendu passe par DOMPurify. Chaque requête de base de données utilise du SQL paramétré. Chaque corps de requête, paramètre de route et chaîne de requête est validé par un schéma Zod à la frontière. Pourquoi c’est important : la plupart des compromissions commencent par une entrée non fiable que l’application n’a pas validée.

4. Demander au navigateur de vous défendre : en-têtes de sécurité. Un Content-Security-Policy, un HSTS d’un an avec préchargement, frame-ancestors none, X-Content-Type-Options nosniff, une politique de référent stricte, et un Permissions-Policy qui désactive les fonctionnalités du navigateur que l’application n’utilise pas. Pourquoi c’est important : même quand l’application se trompe, le navigateur a une seconde ligne de défense. Encore faut-il la lui demander.

5. Ralentir les flots : limites de débit et déni de service. Deux cents requêtes par quinze minutes en ligne de base. Trente par quinze minutes sur la surface d’authentification. Soixante par heure sur les endpoints IA, parce que chaque appel coûte réellement de l’argent. Chaque limite émet un en-tête Retry-After pour que les clients bien élevés se calment proprement. Pourquoi c’est important : un petit attaquant ne devrait pas pouvoir rendre le service inutilisable pour tout le monde.

6. Savoir ce qui s’est passé : journaux et audit. Journaux JSON structurés en production. Chaque requête porte un identifiant de corrélation qui permet de joindre les lignes entre services. Chaque mutation pose une ligne dans une table audit_log avec l’utilisateur, l’IP et les données avant et après. Un caviardage de RP retire les données personnelles avant qu’elles n’atterrissent. Pourquoi c’est important : la première chose que l’on souhaite avoir lors d’un incident, c’est des journaux. La deuxième chose, c’est des journaux qui ne sont pas eux-mêmes une atteinte à la vie privée.

7. Garder les clés en sécurité : secrets et cryptographie. Paires de clés RSA-2048 pour la signature JWT. Clés tenues dans un gestionnaire de secrets (Azure Key Vault, AWS Secrets Manager, GCP Secret Manager, ou Vault). Un environnement validé par Zod qui refuse de démarrer le serveur sur une valeur manquante ou marqueur. Une construction Docker qui échoue si une clé de développement glisse au-delà du fichier ignore. Pourquoi c’est important : une clé qui fuit contourne toutes les autres défenses de cette liste.

8. Garder les données honnêtes : posture de base de données. Une seule piscine de connexions Postgres partagée avec un plafond strict de temps par requête. Des migrations idempotentes (les exécuter deux fois ne change pas le schéma). Une table schema_migration qui suit ce qui a réellement été appliqué. SSL sur le fil vers la base gérée. Pourquoi c’est important : les données sont la partie que l’on ne peut pas refaire. Tout le reste est rejouable.

9. Surveiller vos fournisseurs : chaîne d’approvisionnement. npm audit à chaque push. osv-scanner pour les vulnérabilités hors npm. Une image de base épinglée à la version mineure. Des overrides dans package.json qui épinglent les correctifs CVE transitifs que l’amont n’a pas encore patchés. TruffleHog parcourt l’historique git à la recherche de secrets qui auraient été commités une fois puis oubliés. Pourquoi c’est important : votre code est petit ; vos dépendances sont grandes. Un attaquant n’a qu’à trouver un seul maillon faible.

10. Prévoir quand les choses tournent mal : résilience et reprise. Un script de sauvegarde pg_dump conçu pour cron. Un script de restauration avec une garde contre l’exécution accidentelle en production. Des objectifs de temps et de point de reprise documentés. Des sondes de vivacité et de disponibilité, pour que l’orchestrateur n’envoie du trafic que lorsque la base est joignable. Un arrêt gracieux qui termine les requêtes en cours avant que le pod ne meure. Pourquoi c’est important : les pannes ne sont pas évitables. Elles sont récupérables, à proportion de la préparation faite à l’avance.

11. Savoir que ça fonctionne : tests et évals. Sept cent soixante-treize tests sur quatre-vingt-une suites. Vitest pour les unités. Supertest pour l’intégration HTTP. Playwright pour le bout en bout. Axe pour les violations d’accessibilité. Lighthouse CI pour les budgets de performance. Une vérification mécanique qui s’assure que chaque exigence fonctionnelle prioritaire a une implémentation citée derrière elle. Pourquoi c’est important : on ne peut pas livrer ce qu’on ne peut pas démontrer fonctionner. « Ça marchait sur mon portable » est la phrase la plus coûteuse en logiciel.

12. Garde-fous pour les agents eux-mêmes : sécurité de l’IA. Une interface fournisseur qui permet de changer de fournisseur de modèle par une seule configuration. Un garde-fou de prompt système qui refuse les requêtes hors-périmètre à la frontière. Un caviardage qui passe sur le contenu utilisateur avant tout appel externe. Quatre agents de revue (rouge pour la reconnaissance adversariale, vert pour l’hygiène, bleu pour la revue défensive, jaune pour l’audit de prose) qui auditent le code sur le chemin de la livraison. Pourquoi c’est important : les agents ont leurs propres modes de défaillance. Un agent sans garde est sa propre surface d’attaque.

13. La pile de modèles au moment de l’écriture. Claude Code Opus 4.7 pilote le travail à long terme : la carte de l’application, le modèle de menaces, l’agrégation de la chaîne de destruction, les propositions de correctifs. Le Claude Sonnet 4.6 Agent SDK exécute en parallèle les agents de revue déterministes et les pipelines de reconnaissance. Opus 4.7 tient le contexte de planification d’une boucle de livraison entière dans une seule fenêtre ; le SDK Agent Sonnet 4.6 donne à l’orchestrateur une primitive pour exécuter en parallèle de nombreux petits scanners avec des contrats d’outils explicites. Les contrats du harnais sont indépendants du modèle, donc la même forme fonctionne avec ce qui viendra ensuite.
