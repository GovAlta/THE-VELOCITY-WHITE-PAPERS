Git Insights

We pointed AI at the whole of Alberta's code estate to get the ground truth. Here is how it works and what it found.

Abstract. Alberta holds its code assets in GitHub Enterprise, but the estate had grown into a mire where prototypes mixed with production and no dataset linked to the next. Leadership could not get a definitive answer about what we run or how healthy it is. So, we built Git Insights, an agentic tool that recursively scans the entire estate and reports the ground truth of every repository. 50 agents read 466 million lines of code in roughly 20 hours, a scan no consultant engagement could match. This paper explains the problem, shows in detail how the tool works, and reveals what the scan found. The analysis returned the languages and frameworks we had never counted, the repositories with no tests or documentation, the contributor patterns that quietly degrade security, and the first real benchmark of how AI compares to humans on our own code.
Alberta uses GitHub Enterprise as its primary code repository. It provides powerful insights into code through CodeQL, and integrations with NPM and Maven for dependency vulnerability scanning. Secrets are automatically flagged and blocked so they can be removed, and our cyber team has well-integrated evaluations. The trouble is that these checks are mostly deterministic, logic-based rules, so they carry a high rate of false positives, and they say nothing at all about the things that matter just as much: the absence of documentation, automated build and deployment pipelines, and automated unit tests.


## §01 The data we could not see

The deeper problem we faced was that nothing joined up. Dependency information from GitHub did not flow into our Configuration Management Database. That database lived in ServiceNow and was maintained by hand, with more than 100,000 entries covering these applications. The record of an application was scattered across GitHub, Jira and Confluence tickets, the ServiceNow CMDB, its incidents, SharePoint documentation, and a scatter of spreadsheet trackers. Integrating that data was effectively impossible, because there was no linking key to tie one dataset to the next.

CMDB entries maintained by hand 100,000+. Kept in ServiceNow, with no automated flow from GitHub. The logical structure runs Ministry to Project or Program to Application to repository, and fans out from there into sprints, the CMDB, tickets, and documentation, none of it linked.

The result was that management at the Deputy Minister and Assistant Deputy Minister levels could not get meaningful answers from standard tools. GitHub itself was also at risk of turning into a mire where quick prototypes sat beside production systems with nothing to tell them apart. We could not say with confidence what we ran, how healthy it was, or where the real exposure sat. So we did something about it.


## §02 Building a scout

Using Claude Code as the coding agent, the Opus and Sonnet models for the analysis, and Google Vertex (now the Google Agent Platform) as the agentic layer, Alberta built a tool called **Git Insights**. It is an agentic-first tool that recursively scans the entire GitHub estate. It functions as a ground-truth scout, sent into every repository to come back and report what is actually there. Code forms the ground truth.

Read by roughly 50 agents in about 20 hours 466M lines. A scan of the whole estate that no consultant engagement could match in time or cost. A third of the repositories had no documentation at all, so the agents wrote it, and every dependency was itemized.

The timing matters. Alberta has seen significant increases in the number of known vulnerabilities across the estate, and the most significant inflection in that growth aligns with the release of Anthropic's Mythos, the latest Opus models, and OpenAI's GPT 5.4 and 5.5. The same capability that lets a tool like Git Insights read the estate at depth is what lets attackers find the flaws in it, a point taken up in detail in the [cybersecurity paper](#/paper/mwo98).


## §03 How Git Insights works

Here is what Git Insights actually does, in plain terms. It works in two layers. At the top, a fleet of agents works through the whole estate at once. Inside each repository, every agent runs the same fixed routine. One rule holds it together: the exact, countable work is done by ordinary code that cannot make anything up, and the AI is only ever asked for judgment, in a form we can check against the real code.

The agents run on Google Vertex, and the scan is built to survive the real world. If one repository fails or hits a rate limit, that single job waits and retries while the rest carry on, and if the whole scan is stopped it resumes exactly where it left off. That resilience is what lets it read 466 million lines across thousands of repositories in about 20 hours, rather than the months a manual review would take.

A fair question is how far a government can trust what an AI reports. Three things keep it honest. First, the boring, countable work, listing files and finding the known-bad patterns, is done by ordinary code, which cannot invent a result. Second, the AI is never taken at its word: when it reports a problem it must name the exact file and line, say how serious it is and why, and say how to fix it, so a developer can open the file and confirm the claim. Third, it grades every repository against the same fixed checklist, covering security, code quality, architecture, documentation, tests, maintainability, and the health of the libraries it depends on, so a score means the same thing on the thousandth repository as it did on the first. The output is a finding you can check, not an opinion you have to take on faith.

The same pass does more than grade health. It works out what each system actually does for citizens, detects the old technologies hiding inside, such as COBOL, classic ASP, and long-unsupported versions of .NET and Java, and infers which ministry owns it. It itemizes every dependency a system relies on, which is the exact inventory our Configuration Management Database never had and can now be fed from. It flags the repositories that hold sensitive personal information, so a privacy risk becomes visible rather than buried. And for the 1,280 repositories that had no documentation at all, the agent wrote it from what it had just read, while validating or improving the README on every other repository. That closed the gap to zero: for the first time, every single system in the estate has documentation.

The record it builds for each repository is far deeper than any inventory we had before. It is a full dossier, not a pass or a fail.

All of it lands in one queryable database and a dashboard built for the people who needed answers. There is an executive view of health and risk, a nine-box grid that plots each system by how active it is against how healthy it is, code-health cards per repository, contributor profiles, and the disposition breakdown, all exportable to a document, a spreadsheet, or raw data. The deep cross-repository rollup, where hundreds of overlapping systems in a single ministry are collapsed into a smaller set of rebuilt capabilities, is its own engine and its own paper; see [Git Insights Ministry](#/paper/offjm).

## §04 What the estate actually looked like

With the scan complete, we could finally measure what we had only ever estimated. The domain Git Insights mapped was wider than anyone had counted.

That spread is the case for an opinionated technology stack made in a single table. We are moving toward a small set of supported choices, investing in major platforms such as ServiceNow and the Alberta.ca platform, and eliminating the unique and the underserved. The full argument for that stack is its own paper; see [The Agentic Technology Stack](#/paper/qxlzo).

The structural picture was harder. Across roughly 3,400 repositories, the practices that make code safe to change were missing far more often than they were present.

In aggregate, this analysis was scored to give each repository a health score, rounding to a total score of 10. The composite score across the whole set was not great. 

Average repository health across the estate 3.2 / 10. On the ten-point health scale Git Insights applies to every repository, the estate averaged 3.2. The missing tests, pipelines, and documentation above are most of the reason why.

We also measured the habits of our staff and contractors. We did not use this as a punitive or a performance measure. We measured it to find where ordinary human variance degrades our security controls and best practices, so we know where training and better tooling will help most. Across more than 8,100 contributors and eight years of history, the pattern was clear.

These gaps are not caused malicious actors. They are ordinary, hard-working people working at varying skill levels, and the point of measuring them is that the same patterns which degrade our security controls and best practices are exactly the patterns an AI agent, properly directed, can be set to catch and fix at scale.

Underneath the risk tiers, Git Insights builds a full profile of every contributor, drawn from the record rather than from opinion.


## §05 AI versus human, measured

There is a great deal of criticism aimed at AI for the mistakes it makes, its hallucinations, and there is a strong human recency bias in how those failures get reported: the one wrong answer is remembered, the thousand right ones are not. Viewed in aggregate, and with the proper harnesses described in [the harness paper](#/paper/p7p2k), AI makes significantly fewer mistakes than the criticism assumes.

Git Insights gave us the benchmark to say so with evidence, and it is a benchmark of a kind no one had built before. We measured the real work of more than eight thousand people over eight years: every commit, every pull request, and the state of the code they produced, across the entire estate. We are not aware of a prior analysis of human software work at this scale and over this span anywhere.

With a baseline this complete, we can now say with certainty how AI compares to humans on our own code, measured rather than assumed. Skill levels vary widely, and the mistakes humans make are significant and constant. AI, properly harnessed, makes far fewer of them, and it can be deployed at scale to fix the problems it finds rather than only to count them. We will unpack that comparison in full in a later paper. With this ground truth in hand, the case to invest heavily in the tools, the training, and the harnesses was no longer a matter of faith.

Knowing the estate is the start. The next paper takes Git Insights up a level, to the ministry scale, where hundreds of overlapping systems are read together and collapsed into a smaller set of modern, rebuilt capabilities with a costed plan attached. See [Git Insights Ministry](#/paper/offjm).
